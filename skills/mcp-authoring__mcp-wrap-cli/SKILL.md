# Wrap an existing CLI as an MCP server

The goal: keep the CLI as the canonical surface for humans, and expose a narrow, well-described subset as MCP tools for agents.

## When to use

- User has a battle-tested internal/personal CLI (e.g. a Go TUI, a `gh` extension, a make target wrapper).
- Agents are currently pasting output manually.
- You want deterministic, auditable agent actions (the CLI already logs / has exit codes).

## Process

1. **Inventory subcommands**. List every command / flag combination worth exposing. Discard the rest — agents don't need every admin flag.
2. **Shape tools**. One MCP tool per *intent*, not per subcommand. Example: `pr_ready` (not `gh pr list --state open --label ready`). Tools should be verbs + nouns.
3. **Write JSON Schemas**. Every tool input gets a typed schema. The model will call correctly only when the schema is precise.
4. **Map to execution**. The simplest implementation is `execFile` / `subprocess.run` / `os.Exec`. Capture stdout + stderr, return stdout + exit-code-based error info.
5. **Parse or passthrough**. For tools the agent will reason over, parse JSON from the CLI (prefer `--json`). For tools that are "just run it and tell me what happened," return plain text.

## Native vs wrapper

Two idioms:

**Wrapper MCP** (fastest to ship): A new MCP server (TS or Python) that shells out to the CLI binary. Pros: no CLI code changes. Cons: two artifacts to install.

**Native MCP** (best DX): Add an `mcp` subcommand to the CLI itself (e.g. `gh-tools mcp`), so the binary is both human-facing and agent-facing. Go + [mcp-go](https://github.com/mark3labs/mcp-go) makes this clean.

Pick native when the CLI is already yours and you control releases. Wrapper when the CLI belongs to someone else or you can't modify it.

## Tool description anti-patterns

- ❌ "Runs `gh pr list`" — model doesn't know when to use this.
- ✅ "List open pull requests that are ready for review (`status = open`, label = `ready`). Use when the user asks about pending reviews."

The description is *Claude's routing signal*. Write it as if explaining to a new teammate why you'd pick this tool over alternatives.
